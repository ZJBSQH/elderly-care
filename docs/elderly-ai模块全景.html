概述

elderly-ai 是老年护理云平台的 AI 助手微服务，端口 8087。 它是一个无状态聚合服务——没有自己的数据库，通过 Feign 调用 medicine/remind 服务获取业务数据， 通过千问 DashScope 大模型生成 AI 回答。

本次扩展在原有基础上增加了 RAG 增强检索生成 能力，新增 10 个类，使 AI 能够基于医学知识库给出更精准的回答。

✅ 保留不变：原有 AssistantController 和普通问答链路完全保留，向后兼容
🆕 新增能力：RAG 问答 + 知识库管理，与原有链路并行运行，互不干扰

1.2 两条请求链路

┌── 旧链路（普通问答）─────────────────────┐ │ │ │ POST /ai/medicine/question │ │ │ │ │ ▼ │ │ AssistantController │ │ │ │ │ ├─ MedicalContextBuilder │ │ │ └─ Feign → medicine/remind │ │ │ 拉取用药数据 + 提醒任务 │ │ │ │ │ └─ MedicalAssistant.chat() │ │ └─ 千问大模型 → SSE 流式回答 │ │ │ └──────────────────────────────────────────┘ ┌── 新链路（RAG 增强问答）─────────────────┐ │ │ │ POST /ai/rag/ask │ │ │ │ │ ▼ │ │ RagAssistantController │ │ │ │ │ ▼ │ │ RagService.askWithRag() │ │ │ │ │ ├─ ① 向量检索 │ │ │ └─ EmbeddingModel.embed(问题) │ │ │ └─ RedisEmbeddingStore.search()│ │ │ 返回 Top-5 相似知识片段 │ │ │ │ │ ├─ ② 个人上下文 │ │ │ └─ MedicalContextBuilder │ │ │ (复用旧链路) │ │ │ │ │ ├─ ③ Prompt 增强 │ │ │ └─ 知识 + 个人数据 + 问题 │ │ │ │ │ └─ ④ AI 生成 │ │ └─ MedicalAssistant.chat() │ │ (复用旧链路) │ │ │ └──────────────────────────────────────────┘
设计亮点：新链路复用了旧链路的 MedicalContextBuilder（数据拉取）和 MedicalAssistant（AI 生成）， 只在中间插入了"向量检索 + Prompt 增强"环节。两条链路共享同一套 Feign 客户端和大模型。

5. Feign 客户端

接口目标服务方法路径返回
MedicineFeignClient elderly-medicine (8084) getMedicinesByElderId(elderId) GET /medicine/look/{elderId} Result<List<Map>>
getTodayRecords(elderId) GET /record/today?elderId= Result<List<Map>>
RemindFeignClient elderly-remind (8085) getTodayTasks() GET /remind/task/today Result<List<Map>>
为什么两个 Feign 接口都指定了 configuration = FeignConfig.class？
因为 AI 服务收到前端请求时带有 JWT Token，Feign 调用下游 medicine/remind 服务时也需要携带这个 Token。
FeignConfig 的拦截器自动从 RequestContextHolder 中提取并转发。
