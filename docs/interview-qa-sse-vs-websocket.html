面试模拟 · SSE vs WebSocket

项目：老年护理云平台 | 模块：elderly-ai | 日期：2026-06-05

1. 面试题目

「你的项目里 AI 助手用了 SSE 流式输出。说一下 SSE 是什么？和 WebSocket 有什么区别？为什么选 SSE 而不是 WebSocket？」

2. 常见错误理解

❌ "WebSocket 只能等数据完成后一起展示"

WebSocket 也是实时逐条推送，每条消息客户端立刻就能收到。如果只能最后一起展示，微信网页版、在线聊天室就都没法用了。

❌ "SSE 也是双向通道"

SSE 是单向的——只能服务器推给客户端。客户端要发消息必须另外发起 HTTP 请求。

3. SSE vs WebSocket 核心对比

维度 SSE (Server-Sent Events) WebSocket
方向 ❌ 单向（服务端 → 客户端） ✅ 双向全双工
协议 基于 HTTP，普通请求升级 独立协议 ws:// / wss://
实现复杂度 低 — Spring WebFlux 的 Flux + MediaType.TEXT_EVENT_STREAM 高 — 需要握手处理器、心跳、重连逻辑
自动重连 ✅ 浏览器 EventSource 原生支持 ❌ 需要自己实现
HTTP/2 多路复用 ✅ 兼容 ❌ 不兼容
CDN/反向代理 无需特殊配置 需要代理支持 Upgrade 头
适用场景 AI 流式输出、股票行情、通知推送 即时聊天、在线游戏、协同编辑

4. 为什么 AI 对话选 SSE？

原因一：场景匹配

AI 对话的交互模式：
用户发消息 → 等 AI 回复完 → 用户再发消息 → 等 AI 回复完 → ...

这是典型的"请求-响应"，完全串行。用户不会在 AI 回复到一半时同时发下一条消息。
SSE 的方案：
第1轮：POST /ai/chat（用户消息）→ SSE 流（AI 逐字回复）→ 流结束，HTTP 连接关闭
第2轮：POST /ai/chat（用户第二条消息）→ SSE 流 → 流结束，连接关闭
每一轮"对话 → 回答"都是独立的 HTTP 请求，用完即释放。

原因二：实现简单

Spring Boot WebFlux 中，返回 Flux<ServerSentEvent<String>> 即可实现 SSE 流式输出。前端 uni-app 通过 uni.request + onChunkReceived 回调逐条接收数据。

原因三：基础设施友好

SSE 走普通 HTTP，Nginx、CDN、负载均衡、API 网关不需要任何特殊配置就能透传。WebSocket 需要反向代理开启 Upgrade 支持。

我们的项目实现

// ChatController.java
@PostMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> chatStream(@RequestBody ChatRequest request) {
    return medicalAssistant.chat(request);  // 返回流式 Flux
}

5. WebSocket 的双向能力真正该用在哪里？

WebSocket 强在服务端主动发起推送和双方同时收发互不干扰。

本项目中的正确用法：提醒推送——提醒服务检测到老人该吃药 → 主动推通知到家属手机 → 不需要家属先发请求过来。

这才是真正的"不需要请求"的场景。

6. 常见追问：WebSocket 能不能替代 AI Memory？

错误想法： WebSocket 保持长连接 → 对话历史存在连接对象里 → 不需要 Memory

为什么不行：

  • 连接断了就丢： 用户切后台、刷新页面、网络切换 → 新连接是全新对象 → 历史全部丢失
  • 多实例不共享： 负载均衡到不同服务器 → 各自的内存不互通 → 上下文断裂
  • 跨设备访问做不到： 用户在手机上聊完，想在电脑上继续 → 不同的 WebSocket 连接

AI Memory 必须存在持久化存储（Redis/数据库）里，和连接状态解耦。

7. 面试标准回答

"SSE 是基于 HTTP 的单向流式传输协议，服务端可以持续推送数据，客户端用浏览器原生的 EventSource 就能接收，自动支持重连。

和 WebSocket 的核心区别：SSE 单向，只支持服务端推客户端；WebSocket 双向全双工，双方可以随时互发消息。

AI 对话选 SSE 有三个原因：①场景匹配——用户提问→AI回答是单向流，没有双向需求；②实现简单——Spring WebFlux 的 Flux 天然支持 SSE，几行代码搞定；③基础设施友好——走普通 HTTP，网关/代理不需要特殊配置。

WebSocket 在本项目中用于提醒推送场景——服务端需要主动向客户端推送通知，这才需要双向通道的服务端主动推送能力。"
