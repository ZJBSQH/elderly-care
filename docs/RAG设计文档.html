🐘 老年护理云平台 — RAG 增强检索生成

📅 版本：v1.0 📆 日期：2026-06-07 ✍️ 作者：郑 📦 模块：elderly-ai

文档摘要：本文档描述基于 阿里云通义千问 (DashScope) 嵌入模型 + Redis Stack (RediSearch) 向量数据库的 RAG 增强检索生成方案。 方案在现有 elderly-ai 模块基础上扩展，集成 LangChain4j 框架， 为平台的 AI 助手赋予医学知识库检索能力，提升用药建议、健康问答的准确性和专业性。

1. 概述与目标

1.1 背景

当前 elderly-ai 模块已实现基于千问的流式 AI 问答（AssistantController）， 每次请求通过 MedicalContextBuilder 实时拉取用户的用药数据、提醒任务作为上下文， 再交给大模型生成回答。这种模式存在以下局限：

  • 缺乏专业医学知识库：模型仅依赖自身训练数据，对药品相互作用、禁忌症等专业问题回答不够精准
  • 上下文窗口有限：所有数据塞入 prompt，数据量大时超出 token 限制
  • 知识无法沉淀：平台积累的护理经验、用药指南等无法被 AI 有效利用
  • 幻觉风险：大模型可能对不了解的药品"编造"信息

1.2 目标

#目标优先级
1 建立医学知识向量库，支持药品说明书、护理指南、健康知识的语义检索 P0 核心
2 RAG 增强问答：检索相关文档片段 → 注入 prompt → 生成更精准的回答 P0 核心
3 知识库管理 API：支持文档的上传、删除、更新、列表查询 P1 重要
4 支持增量索引：新增/更新文档时自动重新嵌入并更新向量库 P1 重要
5 检索结果可追溯：返回引用来源，增强用户信任 P2 优化

属性说明

4.3 向量检索器

/**
 * 检索器配置
 */
@Component
public class RagRetrieverConfig {

    @Bean
    public EmbeddingStoreRetriever embeddingStoreRetriever(
            RedisEmbeddingStore embeddingStore,
            DashScopeEmbeddingModel embeddingModel) {

        return EmbeddingStoreRetriever.builder()
                .embeddingStore(embeddingStore)      // Redis 向量存储
                .embeddingModel(embeddingModel)      // 千问嵌入模型
                .maxResults(5)                       // 返回 Top-5 最相似片段
                .minScore(0.65)                      // 最低相似度阈值（COSINE）
                .build();
    }
}

检索参数说明

参数默认值说明
maxResults5返回最相似的前 K 个片段，值越大上下文越丰富但 token 消耗越多
minScore0.65余弦相似度最低阈值，低于此值的片段丢弃（避免引入噪声）
distanceMetricCOSINE余弦距离，适合语义相似度比较

4.4 RAG 增强问答

RagService 核心逻辑

/**
 * RAG 增强问答服务
 * 

 * 核心流程：用户问题 → 向量检索 → Prompt 组装 → AI 生成 → 流式输出
 */
@Service
@RequiredArgsConstructor
@Slf4j
public class RagService {

    private final EmbeddingStoreRetriever retriever;
    private final MedicalAssistant assistant;         // 复用现有 AI Service
    private final MedicalContextBuilder contextBuilder; // 复用现有上下文构建

    /**
     * RAG 增强问答（SSE 流式输出）
     *
     * @param elderId  老人ID（用于个性化上下文）
     * @param question 用户问题
     * @return 流式 AI 回答
     */
    public Flux<String> askWithRag(Integer elderId, String question) {

        // 步骤1: 检索相关知识片段
        List<TextSegment> relevantDocs = retriever.findRelevant(question);
        log.info("检索到 {} 个相关片段", relevantDocs.size());

        // 步骤2: 构建个性化医疗上下文（复用现有逻辑）
        String personalContext = contextBuilder.buildAndFormat(elderId);

        // 步骤3: 组装增强 Prompt
        String augmentedPrompt = buildAugmentedPrompt(
                personalContext, relevantDocs, question);

        // 步骤4: 流式生成回答
        return assistant.chat(augmentedPrompt);
    }

    /**
     * 组装增强 Prompt（核心）
     * 

     * 结构: System指令 → 检索知识 → 个人上下文 → 用户问题
     */
    private String buildAugmentedPrompt(
            String personalContext,
            List<TextSegment> docs,
            String question) {

        StringBuilder sb = new StringBuilder();
        sb.append("【参考知识】请基于以下专业知识回答用户问题：\n");
        for (int i = 0; i < docs.size(); i++) {
            TextSegment doc = docs.get(i);
            sb.append(String.format("--- 参考资料%d (来源: %s) ---\n%s\n",
                    i + 1,
                    doc.metadata().getOrDefault("title", "未知"),
                    doc.text()));
        }
        sb.append("\n【用户个人数据】\n").append(personalContext);
        sb.append("\n【用户问题】").append(question);
        sb.append("\n\n请结合参考知识和用户个人数据，给出专业、安全、易懂的回答。");
        sb.append("如果参考知识不足以回答，请明确说明。");
        return sb.toString();
    }
}

4.6 Redis 向量存储 Schema 设计

索引定义

// Redis Stack FT.CREATE 索引定义
FT.CREATE idx:rag_docs           // 索引名称
  ON HASH                         // 基于 Hash 数据结构
  PREFIX 1 rag:doc:               // Key 前缀过滤
  SCHEMA
    content      TEXT              // 原始文本（支持关键词搜索）
    embedding    VECTOR HNSW       // 向量字段
      6
      TYPE FLOAT32
      DIM 1536                     // 与嵌入模型输出维度一致
      DISTANCE_METRIC COSINE       // 余弦距离
    title        TAG               // 标题（精确匹配过滤）
    category     TAG               // 分类（按分类过滤用）
    chunk_index  NUMERIC           // 片段序号
    created_at   NUMERIC           // 创建时间戳
    doc_id       TAG               // 文档ID（用于删除/更新）

Key 命名规范

Key 模式示例说明
rag:doc:{docId}:chunk:{n}rag:doc:a1b2c3:chunk:0文档的第 n 个片段
rag:doc:{docId}:metarag:doc:a1b2c3:meta文档元信息（title, category 等）
rag:doc:ids—SET 类型，存储所有文档 ID 集合

      max-overlap: 50          # 重叠字符数
    # 检索配置
    retrieval:
      max-results: 5           # 默认检索 Top-K
      min-score: 0.65          # 最低相似度阈值
    # 知识库配置
    knowledge-base:
      # 支持的文件类型
      allowed-types: txt,md,json
      # 单文件最大大小
      max-file-size: 5MB

8. 风险与注意事项

风险影响应对策略
嵌入 API 调用失败 文档入库中断 / 问答无法检索 ① 添加重试机制（3次指数退避）
② 降级为纯大模型回答（无 RAG）
Redis Stack 内存溢出 检索超时或服务不可用 ① 设置 maxmemory 限制
② 监控向量索引大小
③ 定期清理过期文档
嵌入成本过高 大量文档入库时 API 费用攀升 ① 批量嵌入合并请求
② 缓存已嵌入的文档片段
③ 按需入库（非全量同步）
检索噪声 不相关片段污染 Prompt，降低回答质量 ① 合理设置 minScore 阈值
② 可引入 Reranker 二次排序
③ Prompt 中明确"参考资料不足时请说明"
医学信息过时 过时药品信息误导用户 ① 文档元数据记录版本号和更新日期
② 定期审核知识库内容
③ 回答中添加时效性声明
Redis 标准版升级到 Stack 不兼容导致现有缓存失效 ① 在测试环境先验证
② 保留原有 RDB 备份
③ 切换后回归测试 RedisUtil
📌 总 结

本方案充分利用项目现有基础设施（千问 DashScope、Redis、LangChain4j），以最小成本引入 RAG 能力：
  • 嵌入层：复用已配置的千问 text-embedding-v3 模型
  • 存储层：Redis 升级到 Stack 版本，一个镜像替换即可
  • 框架层：新增 1 个 LangChain4j Redis starter 依赖
  • 业务层：新增 ~10 个 Java 类，与现有 MedicalAssistant 无缝协作
预计开发周期：3-5 个工作日（含测试和文档入库）。
