📌 1.1 什么是 RAG（检索增强生成）

RAG = Retrieval Augmented Generation（检索增强生成），核心思想是：给大模型"开卷考试"而非"闭卷作答"。

用户问题: "布洛芬和阿司匹林能一起吃吗？" │ ▼ ① 向量化 ─── EmbeddingModel.embed(question) ──→ 1024维向量 │ ▼ ② 检索 ──── EmbeddingStore.search(queryVector) ──→ Top-K 相似知识片段 │ (来自医学知识库) ▼ ③ 增强 ──── 参考知识 + 用户数据 + 问题 ──→ 组装增强Prompt │ ▼ ④ 生成 ──── ChatModel.generate(augmentedPrompt) ──→ 有据可依的AI回答
RAG解决了LLM三大痛点：
① 幻觉 — 模型凭空编造，RAG用真实文档约束回答
② 知识过时 — 模型训练数据有截止日期，RAG可检索最新文档
③ 领域盲区 — 通用模型不了解私有医学知识，RAG补充专属知识库

🧩 1.2 LangChain4j 中的 RAG 核心组件

组件接口/类作用常见实现
嵌入模型 EmbeddingModel 将文本转为向量 DashScope / OpenAI / HuggingFace / Ollama

🔀 1.3 两种集成模式对比

模式一：Naive RAG（ContentRetriever）

// ✅ 简单直接，快速上手
EmbeddingStore<TextSegment> embeddingStore = ...;
EmbeddingModel embeddingModel = ...;

ContentRetriever contentRetriever = EmbeddingStoreContentRetriever.builder()
    .embeddingStore(embeddingStore)
    .embeddingModel(embeddingModel)
    .maxResults(5)
    .minScore(0.75)
    .build();

    .chatModel(chatModel)
    .contentRetriever(contentRetriever)
    .build();

模式二：Advanced RAG（RetrievalAugmentor）

// ✅ 功能更强：查询变换、重排序、自定义注入策略
RetrievalAugmentor retrievalAugmentor = DefaultRetrievalAugmentor.builder()
    .queryTransformer(...)      // 查询改写（扩展、多角度）
    .queryRouter(...)           // 路由到不同知识库
    .contentAggregator(...)     // 多路结果聚合
    .contentInjector(...)       // 自定义Prompt注入格式
    .executor(...)              // 线程池配置
    .build();

Assistant assistant = AiServices.builder(Assistant.class)
    .chatModel(chatModel)
    .retrievalAugmentor(retrievalAugmentor)
    .build();
选择建议：项目初期用 Naive RAG（ContentRetriever），需要查询改写、重排序或多知识库路由时升级到 Advanced RAG。两者不互斥，可以在不同 @AiService 接口中混用。

🗄️ 2.4 EmbeddingStore 接口详解

这是 RAG 中最核心的接口——所有向量数据库集成必须实现此接口。理解它是自定义向量存储的基础。

完整接口定义

public interface EmbeddingStore<Embedded> {

    // ═══════════ 写入操作 ═══════════
    String  add(Embedding embedding);
    void    add(String id, Embedding embedding);
    String  add(Embedding embedding, Embedded embedded);
    List<String> addAll(List<Embedding> embeddings);
    List<String> addAll(List<Embedding> embeddings, List<Embedded> embedded);

    // ═══════════ 删除操作 ═══════════
    void remove(String id);
    void removeAll(Collection<String> ids);
    void removeAll(Filter filter);  // 按条件删除
    void removeAll();                // 清空

    // ═══════════ 检索操作（核心） ═══════════
    EmbeddingSearchResult<Embedded> search(EmbeddingSearchRequest request);
}

EmbeddingSearchRequest / EmbeddingSearchResult

// 检索请求
EmbeddingSearchRequest request = EmbeddingSearchRequest.builder()
    .queryEmbedding(queryEmbedding)  // 查询向量（由 EmbeddingModel 生成）
    .maxResults(5)                   // 返回 Top-K
    .minScore(0.65)                  // 最低相似度阈值（0~1）
    .filter(...)                     // 可选：按元数据过滤
    .build();

// 检索结果
EmbeddingSearchResult<TextSegment> result = embeddingStore.search(request);
List<EmbeddingMatch<TextSegment>> matches = result.matches();

for (EmbeddingMatch<TextSegment> match : matches) {
    String id = match.embeddingId();      // 向量记录 ID
    double score = match.score();          // 相似度 (0~1)
    Embedding embedding = match.embedding(); // 原始向量
    TextSegment segment = match.embedded();  // 文本片段（含元数据）
}

✂️ 3.1 文档切分（Document Splitting）

递归切分策略

// DocumentSplitters.recursive() 的切分层级：
// 段落（\n\n）→ 行（\n）→ 句子（。！？）→ 词（空格）→ 字符

var splitter = DocumentSplitters.recursive(500, 50);
//                                          ↑     ↑
//                                    最大字符数  重叠字符数

// 切分流程
Document document = Document.from("这是一篇很长的医学文章...");
List<TextSegment> segments = splitter.split(document);

// 为每个片段注入元数据
for (int i = 0; i < segments.size(); i++) {
    TextSegment seg = segments.get(i);
    seg.metadata().put("chunk_index", i);
    seg.metadata().put("doc_id", docId);
}

切分策略对比

策略方法适用场景
递归切分recursive(max, overlap)✅ 通用文本（推荐）
按段落切分byParagraph(max, overlap)结构清晰的长文
按句子切分bySentence(max, overlap)短句密集型文本
按字符切分fixed(max, overlap)无结构文本
医学文档建议：药品说明书的【适应症】【用法用量】【禁忌】等短句密集段落在 500 字符的 chunk 中可能混杂多个主题。可以考虑缩小 max-size 到 300，或使用按段落切分后再用按句子切分的组合策略。

🧮 3.2 向量嵌入（Embedding）

// 单条嵌入
Embedding embedding = embeddingModel.embed("布洛芬缓释胶囊").content();
float[] vector = embedding.vector();  // 1024 维 float 数组

// 批量嵌入（一次 HTTP 请求提交所有片段）
List<TextSegment> segments = splitter.split(document);
List<Embedding> embeddings = embeddingModel.embedAll(segments).content();
//                             ↑ 注意：embedAll 是一次性全量提交！
embedAll() 的风险：
① 一次性提交：100 个片段 = 1 个 HTTP 请求，超时风险高
② 全有或全无：API 返回中途失败时，不知道哪些成功了
③ 无断点续传：失败后需要从头重试

改进方案 → 分批嵌入： 
// ✅ 分批嵌入（推荐）
private List<Embedding> embedBatch(List<TextSegment> segments, int batchSize) {
    List<Embedding> all = new ArrayList<>();
    for (int i = 0; i < segments.size(); i += batchSize) {
        int end = Math.min(i + batchSize, segments.size());
        List<TextSegment> batch = segments.subList(i, end);
        try {
            all.addAll(embeddingModel.embedAll(batch).content());
            log.info("嵌入进度: {}/{}", end, segments.size());
        } catch (Exception e) {
            log.error("嵌入批次 [{}, {}) 失败: {}", i, end, e.getMessage());
            // ← 可以重试，而非全量重来
        }
    }
    return all;
}

    TextSegment segment = match.embedded();  // 文本片段

}

📝 3.4 Prompt 增强（Augmented Prompt）

增强 Prompt 的标准结构

// 三种信息源的融合
public String buildAugmentedPrompt(
        List<EmbeddingMatch<TextSegment>> knowledgeDocs,  // 知识库检索结果
        String personalContext,                              // 用户个人数据
        String userQuestion) {                               // 用户原始问题

    StringBuilder sb = new StringBuilder();

    // ① 参考知识（来自向量知识库）
    sb.append("【参考知识】请基于以下专业知识回答用户问题，");
    sb.append("如果参考知识不足以回答，请明确说明：\n");
    for (int i = 0; i < knowledgeDocs.size(); i++) {
        TextSegment seg = knowledgeDocs.get(i).embedded();
        String source = seg.metadata().getString("title");
        double score = knowledgeDocs.get(i).score();
        sb.append(String.format(
            "--- 参考资料%d (来源: %s, 相关度: %.2f) ---\n%s\n",
            i + 1, source, score, seg.text()));
    }

    // ② 用户个人数据（来自业务系统）
    sb.append("\n【用户个人数据】\n").append(personalContext);

    // ③ 用户问题
    sb.append("\n【用户问题】").append(userQuestion);
    sb.append("\n\n请结合参考知识和用户个人数据，给出专业、安全、易懂的回答。");
    sb.append("涉及用药安全时必须谨慎，建议咨询医生。");

    return sb.toString();
}

Prompt 模板最佳实践

区块内容作用
【参考知识】向量检索的 Top-K 片段 + 来源标注约束回答范围，减少幻觉
【用户个人数据】药品清单、用药记录、提醒任务个性化上下文
【用户问题】原始自然语言问题核心任务
【输出约束】不使用Markdown、安全警告格式控制 + 责任规避

⚠️ 4.1 异常分层模型（核心）

RAG 管道涉及 5 个独立的服务边界，每一层都需要独立的异常处理和降级策略。 最常见的错误是"静默失败"——捕获异常后返回空列表，调用方无法区分"没数据"和"服务挂了"。
askWithRag(elderId, question) ← 最外层 try-catch（降级入口） │ ├─ ① embeddingModel.embed(question) ← 网络/认证异常 │ └─ DashScope API 不可达 / API Key 无效 │ ├─ ② embeddingStore.search(request) ← 连接/索引/数据异常 │ └─ Redis 断开 / 索引不存在 / 格式错误 │ ├─ ③ contextBuilder.buildAndFormat() ← Feign 调用异常 │ └─ medicine/remind 服务不可达 / 超时 │ └─ ④ assistant.chat(augmentedPrompt) ← AI 模型异常 └─ 千问 API 超时 / 限流 / Token超限

异常分类策略

异常类型是否可恢复处理策略用户感知
不可恢复 认证失败（API Key无效） ❌ 快速失败 + 告警 "服务暂时不可用"
可重试 网络超时 ✅ 重试 2~3 次后降级 无感知
可降级 检索失败 ✅ 跳过RAG，纯LLM回答 回答缺少知识库支撑
可降级 Feign 调用失败 ✅ 使用空数据占位 回答缺少个人化数据
可降级 AI 模型不可用 ❌ 返回友好友好错误 "请稍后再试"

🧮 4.2 嵌入模型异常（EmbeddingModel）

异常场景

// 场景1：网络超时（最常见）
// 场景2：API Key 无效或过期
// 场景3：模型名称不存在
// 场景4：返回的向量维度与预期不一致

Embedding queryEmbedding;
try {
    queryEmbedding = embeddingModel.embed(question).content();
} catch (RuntimeException e) {
    log.error("嵌入模型调用失败: {}", e.getMessage(), e);
    // ← 嵌入是 RAG 的第一步，失败意味着整个 RAG 不可用
    // 应该直接降级到纯 LLM 回答
    throw new RagUnavailableException("嵌入服务不可用", e);
}

正确处理模式

/**
 * 嵌入异常 → 整个 RAG 不可用 → 降级为纯 LLM
 * 注意：embedAll() 批量嵌入的超时风险更高
 */
private List<EmbeddingMatch<TextSegment>> retrieveRelevant(String question) {
    try {
        Embedding queryEmbedding = embeddingModel.embed(question).content();
        // 校验向量维度
        if (queryEmbedding.vector().length != this.dimension) {
            log.error("向量维度不匹配: 期望{}, 实际{}",
                this.dimension, queryEmbedding.vector().length);
            throw new IllegalStateException("嵌入模型维度配置错误");
        }
        return embeddingStore.search(
            EmbeddingSearchRequest.builder()
                .queryEmbedding(queryEmbedding)
                .maxResults(maxResults)
                .minScore(minScore)
                .build()
        ).matches();
    } catch (Exception e) {
        // ← 统一向上抛，由外层 askWithRag() 统一降级
        log.error("向量检索失败，将降级为纯大模型回答", e);
        throw new RuntimeException("RAG检索不可用", e);
    }
}
关键原则：retrieveRelevant() 应该统一抛异常，而非内部吞掉。 如果内部 catch 后返回空列表，外层无法区分"没有匹配结果"和"服务挂了"。 两种场景的降级行为可能不同——前者正常返回（只是没有知识增强），后者需要告警。

@Override
public EmbeddingSearchResult<TextSegment> search(EmbeddingSearchRequest request) {

        // ... 执行检索 ...

    } catch (JedisConnectionException e) {
        // ① 连接失败 → 可重试，重试失败后向上抛
        log.error("Redis 连接失败: {}", e.getMessage());
        throw new RuntimeException("向量存储不可用，请检查 Redis Stack 状态", e);
        //   ↑ 不要返回空列表！让调用方走降级

    } catch (JedisDataException e) {
        // ② 索引不存在 → 尝试自动重建
        if (e.getMessage().contains("no such index")) {
            log.warn("索引 {} 不存在，尝试重建", indexName);
            createIndex();
            return new EmbeddingSearchResult<>(List.of());
        }
        throw e;

    } catch (NumberFormatException e) {
        // ③ Score 解析异常 → 数据损坏，跳过该条
        log.error("Score 解析失败，可能向量数据损坏: {}", e.getMessage());
        // 返回已解析的成功部分（降级返回）
    }

    return new EmbeddingSearchResult<>(matches);
}

// ✅ 正确：根据异常类型决定是否重试
private void addInternal(String id, Embedding embedding, TextSegment segment) {
    int retries = 0;
    int maxRetries = 3;
    while (retries < maxRetries) {
        try {
            jedis.hset(key, fields);
            return;  // 成功
        } catch (JedisConnectionException e) {
            retries++;
            if (retries == maxRetries) {
                throw new RuntimeException(
                    String.format("写入失败（已重试%d次）: %s", maxRetries, id), e);
            }
            log.warn("写入重试 {}/{}: {}", retries, maxRetries, id);
            try { Thread.sleep(1000L * retries); } catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("写入被中断", ie);
            }
        } catch (Exception e) {
            throw new RuntimeException("写入失败: " + id, e);
        }
    }
}

@Override
public List<String> addAll(List<Embedding> embeddings, List<TextSegment> textSegments) {
    // 数量一致性检查
    if (embeddings.size() != textSegments.size()) {
        throw new IllegalArgumentException(
            String.format("嵌入向量数量(%d)与文本片段数量(%d)不一致",
                embeddings.size(), textSegments.size()));
    }
    // 向量维度检查
    for (int i = 0; i < embeddings.size(); i++) {
        if (embeddings.get(i).vector().length != this.dimension) {
            throw new IllegalArgumentException(
                String.format("第%d个向量维度不匹配: 期望%d, 实际%d",
                    i, this.dimension, embeddings.get(i).vector().length));
        }
    }
    // ... 执行写入 ...
}
写入失败的断点续传问题：如果 100 个片段中第 50 个写入失败，前面的 49 个已经写入了。 简单的重试会导致前 49 个重复写入。需要在设计层面考虑：① 写入前先按 doc_id 清理旧数据 ② 或者记录每个 chunk 的写入状态（Redis Set）

🤖 4.6 AI 调用异常 + 完整降级链路

最终降级策略

/**
 * RAG 增强问答——完整降级链路
 *
 * 降级层级：
 *   L1: RAG 全部可用 → 知识库 + 个人数据 + AI
 *   L2: 检索失败     → 仅个人数据 + AI（本方法）
 *   L3: 个人数据失败  → 仅 AI（纯聊天）
 *   L4: AI 不可用     → 返回固定话术
 */
public Flux<String> askWithRag(Integer elderId, String question) {
    try {
        // L1: 正常 RAG 流程
        List<EmbeddingMatch<TextSegment>> docs = retrieveRelevant(question);
        String personalCtx = contextBuilder.buildAndFormat(elderId);
        String prompt = buildAugmentedPrompt(docs, personalCtx, question);
        return assistant.chat(prompt)
            .onErrorResume(e -> {
                // L4 兜底：AI 模型也不可用
                log.error("AI 调用失败", e);
                return Flux.just("抱歉，AI 服务暂时不可用，请稍后再试。");
            });

    } catch (Exception e) {
        // L2: RAG 检索失败 → 降级为纯个人数据 + AI
        log.warn("RAG 检索失败，降级为个人数据问答", e);
        try {
            String personalCtx = elderId != null
                ? contextBuilder.buildAndFormat(elderId)
                : "（无个人数据）";
            return assistant.chat(personalCtx + "\n\n用户问题: " + question)
                .onErrorResume(ex -> {
                    // L3: 个人数据也获取失败 → 纯 AI 聊天
                    log.error("上下文构建失败", ex);
                    return assistant.chat(question);
                });
        } catch (Exception ex) {
            // L4: 全部失败
            log.error("所有路径均失败", ex);
            return Flux.just("抱歉，服务暂时不可用，请稍后再试。");
        }
    }
}
降级设计原则：
① 逐层降级：RAG → 个人数据 → 纯AI → 固定话术，每一层都是上一层的兜底
② 用户无感第一层：检索失败时用户感知不到（只是回答少了知识增强）
③ 清晰告知最后一层：AI 完全不可用时，必须明确告诉用户"请稍后再试"
④ 日志完整：每一层降级都要打 log，便于排查根因

// float[] → FLOAT32 小端字节数组（RediSearch 标准格式）
private byte[] embeddingToBytes(Embedding embedding) {
    float[] vector = embedding.vector();
    ByteBuffer buffer = ByteBuffer.allocate(vector.length * 4);
    buffer.order(ByteOrder.LITTLE_ENDIAN);  // ← 小端序
    for (float v : vector) {
        buffer.putFloat(v);
    }
    return buffer.array();
}

// 字节数组 → RediSearch BLOB 字符串
private String bytesToBlob(byte[] bytes) {
    return new String(bytes, StandardCharsets.ISO_8859_1);
    //                    ↑ ISO_8859_1: 每个字节映射为一个字符，无数据丢失
}

🛡️ 5.2 降级策略设计（完整实战）

降级链路图

用户请求 → askWithRag(elderId, question) │ ├─ [成功] retrieveRelevant(question) ─→ Top-K 知识片段 │ │ │ └─ [失败] ─→ 降级①: 跳过RAG，仅用个人数据 + AI │ │ │ └─ [失败] ─→ 降级②: 纯AI聊天（无上下文） │ │ │ └─ [失败] ─→ 降级③: 固定话术 │ └─ [成功] contextBuilder.buildAndFormat(elderId) ─→ 个人上下文 │ └─ [失败] ─→ 降级④: "（无个人数据）" 占位符

关键——降级行为必须一致

// ❌ 不一致的设计（当前代码的问题）
private List<EmbeddingMatch<TextSegment>> retrieveRelevant(String question) {
    Embedding queryEmbedding = embeddingModel.embed(question).content();
    // ↑ 如果这里抛异常 → 外层 catch 捕获 → 走降级 ✅

    return embeddingStore.search(request).matches();
    // ↑ 如果这里抛异常 → 被 search() 内部吞掉 → 返回空列表 → 不走降级 ❌
}

// ✅ 一致的设计
// retrieveRelevant() 统一向上抛异常，由外层统一降级
private List<EmbeddingMatch<TextSegment>> retrieveRelevant(String question) {
    // 所有异常都不吞，直接向上传播
    Embedding queryEmbedding = embeddingModel.embed(question).content();
    return embeddingStore.search(request).matches();
    // embeddingStore.search() 也不吞异常
}
降级设计检查清单：
☑ 每一层异常都有明确的降级路径
☑ 降级行为在全链路中保持一致（不会一半走降级一半不走）
☑ 最底层降级（完全不可用）必须给用户友好提示
☑ 每一层降级都有 log.warn/log.error
☑ 降级后的回答质量仍然可接受（不会给出错误信息）

👂 5.3 EmbeddingStoreListener（监听器模式）

LangChain4j 提供了 EmbeddingStoreListener 接口，可以在写入/检索时自动触发回调，用于监控、日志、审计等——无需修改业务代码。 
// 注册监听器
EmbeddingStore<TextSegment> store = new RedisEmbeddingStore(...);

// addListener 返回包装后的 EmbeddingStore（装饰器模式）
EmbeddingStore<TextSegment> monitoredStore = store.addListener(
    new EmbeddingStoreListener<TextSegment>() {

        @Override
        public void onAdd(String id, Embedding embedding, TextSegment segment) {
            log.info("向量入库: id={}, title={}", id,
                segment != null ? segment.metadata().getString("title") : "N/A");
        }

        @Override
        public void onSearch(EmbeddingSearchRequest request,
                             EmbeddingSearchResult<TextSegment> result) {
            log.info("检索完成: maxResults={}, 实际返回={}",
                request.maxResults(), result.matches().size());
        }

        @Override
        public void onError(EmbeddingStoreErrorContext<TextSegment> context) {
            Throwable error = context.error();
            log.error("向量存储异常: {}", error.getMessage());
            // 可以在这里接入告警系统
            alertService.send("RAG向量存储异常: " + error.getMessage());
        }
    }
);

维度EmbeddingStoreListenertry-catch

🚨 6.1 10 大反模式（避坑指南）

#反模式问题正确做法
1 静默失败
catch(Exception e) { return List.of(); }		 调用方无法区分"没数据"和"服务挂了"，降级逻辑被绕过 异常向上传播，由顶层统一降级
2 双层catch行为不一致
内层吞异常，外层走降级		 同一方法内，embed失败走降级，search失败不走降级 统一策略：要么都吞，要么都抛
3 embedAll()不分批
100个片段 = 1个HTTP请求		 超时风险、部分失败不可恢复、无断点续传 分批嵌入（每批10~20个）+ 记录进度
4 Feign返回裸Map
Result<List<Map>>		 泛型擦除、零编译期安全、运行时ClassCastException 定义类型化DTO
5 Event Loop线程阻塞
同步Feign调用不用defer		 阻塞Netty事件循环 → 吞吐量暴跌 Flux.defer()包裹阻塞操作
6 空实现充当已实现
return List.of();		 调用方以为查询成功但无数据（契约欺骗） 抛UnsupportedOperationException
7 SSE格式不一致
有的有[DONE]有的没有		 前端处理流结束的逻辑分裂 统一SSE事件格式
8 安全白名单过宽
permitAll("/ai/**")		 知识库管理接口完全暴露，未认证即可增删改 细化白名单，区分读写权限
9 测试兜底泄露到生产
if(id==null) id=1;		 未登录用户使用测试账号数据 → 信息泄露 加TODO标记 + 上线前移除
10 格式化输出违反SystemMessage约束
SystemMessage禁止Markdown但输出使用Markdown		 Prompt约束与实际输出格式冲突 在约束框架内设计格式（纯文本友好）
核心原则：以上 10 条反模式中，#1（静默失败）和#2（双层catch不一致）是最容易犯也最致命的——它们直接破坏降级策略的正确性。修复这两条就能避免 80% 的异常处理相关问题。
