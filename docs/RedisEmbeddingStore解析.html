前言：这个类是干什么的

RedisEmbeddingStore 就是一个"智能文档仓库"——它能把医学知识文档存进 Redis， 并且支持"用一句话搜出最相关的文档片段"。

想象你有一个超级图书管理员：
① 存书时，他把每本书拆成短段落，给每段贴上一个"含义标签"（向量），然后归档
② 找书时，你问他一句话，他扫描所有标签，找到含义最接近的几段还给你
这个类就是那个管理员的大脑。
能力对应方法一句话解释
存add() / addAll()把向量 + 文本存进 Redis Hash
删remove() / removeByDocId()按 ID 或按文档批量删除
查search() ⭐用一句话找到最相似的 Top-K 片段
建库createIndex()在 Redis 中建立 HNSW 向量索引

1. 宏观视角：数据如何流转

先看全局，再看细节。从"文档上传"到"智能问答"，数据经历了以下旅程：

📄 医学文档（比如"布洛芬说明书"） │ ▼ IngestionPipeline.split() ┌──────────────────────┐ │ 切成 N 个小片段 │ 每段 ~500 字符，有重叠 │ 片段0: "布洛芬用于缓解..." │ │ 片段1: "...轻至中度疼痛，如..." │ │ 片段2: "...头痛、牙痛、痛经..." │ └──────┬───────────────┘ │ ▼ IngestionPipeline.embed() ← 调千问 API ┌──────────────────────┐ │ 每个片段 → 1536 维向量 │ 向量 = 含义的数学表示 │ 片段0 → [0.12, -0.34, ...] │ │ 片段1 → [0.08, 0.41, ...] │ └──────┬───────────────┘ │ ▼ RedisEmbeddingStore.addAll() ← 本类接管！ ┌──────────────────────┐ │ Redis Hash │ │ Key: rag:doc:uuid1 │ │ { │ │ content: "布洛芬...",│ │ embedding: <6144字节>,│ │ title: "布洛芬说明书", │ │ category: "MEDICINE", │ │ ... │ │ } │ └──────────────────────┘ │ │ ... 用户提问"头痛该吃什么药？" ... │ ▼ RedisEmbeddingStore.search() ← 核心！ ┌──────────────────────┐ │ 1. 问题 → 向量 │ │ 2. KNN 向量检索 │ 找 Top-5 最相似 │ 3. 返回匹配的文本片段 │ └──────┬───────────────┘ │ ▼ RagService ┌──────────────────────┐ │ 检索结果 + Prompt → AI │ 千问大模型生成答案 └──────────────────────┘
本类负责的是红色虚线框内的部分：向量的存储、检索、删除。它不负责"切分"和"嵌入"——那是 IngestionPipeline 的工作。

2. 核心概念扫盲

2.1 什么是"向量"？

假设你要用 3 个数字描述一个人：身高(175)、体重(70)、年龄(30) → [175, 70, 30] 就是一个三维向量。

大模型做的事更厉害：用 1536 个数字 描述一段文字的含义。两段含义相近的文字，它们的 1536 个数字也"长得像"——这就是语义搜索的基础。

"布洛芬可以缓解头痛" 和 "布洛芬用于镇痛" → 向量很接近（含义相似）
"布洛芬可以缓解头痛" 和 "今天天气不错" → 向量相距很远（含义不相关） 

// 一段文字 → 千问 embedding API → 1536 个 float
float[] vector = {0.12, -0.34, 0.08, 0.41, ... /* 共 1536 个 */};

2.2 什么是 KNN？

KNN = K-Nearest Neighbors = K 个最近的邻居
你在一个大厅里有 1000 个人，每个人拿着一张纸，上面写着一段医学知识。

现在你喊了一句话："头痛吃什么药？"

KNN 做的事：在内存中计算你说的这句话和 1000 张纸上内容"含义有多接近"，然后挑出最接近的 K=5 张纸给你。

这个"含义有多接近"，在数学上就是两个向量之间的距离。

在代码中，KNN 体现在这行：

// KNN 查询语法（RediSearch 方言）
String query = "*=>[KNN " + maxResults + " @embedding $BLOB AS score]";
//                    ^^^^^^^^
//                    这里的 K = 5（默认返回 5 个最相似结果）
参数含义调大了会怎样调小了会怎样
K = 5返回最相似的 5 个片段更多参考信息，但耗 token更精准，但可能遗漏
minScore = 0.65相似度低于 65% 的不要返回更多（含噪声）更严格，可能无结果

2.3 什么是 HNSW？

HNSW = Hierarchical Navigable Small World = 分层可导航小世界图
没有 HNSW 的搜索（暴力扫描）：
你要在 10000 本书里找"最像的一本"，需要一本一本翻——O(N)，极慢。

有 HNSW 的搜索：
你先看"大类标签"（顶层），找到合适的大类；再看"小类标签"（中层）；最后在几本书里比较（底层）。
每层都是一个"高速公路网"，可以快速跳转到目标区域——O(log N)，极快。

就像 GPS 导航：你不会从北京一步步走到上海，而是先上高速 → 再走省道 → 最后走街道。

在索引创建时，这几个参数决定了 HNSW 的行为：

vectorAttrs.put("EF_CONSTRUCTION", 200);  // 建图时的搜索宽度——越大越精确，建得越慢
vectorAttrs.put("M", 16);                 // 每个节点的最大连接数——越大越精确，内存越多
// DIM 1536                                // 向量的维度，由 embedding 模型决定

2.4 什么是 COSINE 相似度？

两个向量的 COSINE = 它们"方向"有多一致（不看长度）。

你的观点和我的观点有多"合拍"，不取决于我们说话声音多大（长度），而取决于我们的观点方向是不是一致（角度）。

COSINE = 1.0 → 方向完全相同（含义一模一样）
COSINE = 0.0 → 完全无关
COSINE = -1.0 → 含义相反

EmbeddingSearchRequest 是 LangChain4j 定义的标准请求对象，封装了"搜什么""返回几个""最低多像"三个信息。

4.2 问题向量化：把"文字"变成"数字" L192-193

    // 将查询向量序列化为字节
    byte[] queryVector = embeddingToBytes(request.queryEmbedding());

这里的 request.queryEmbedding() 是调用方（RagService）已经通过千问 API 把用户问题转换好的 1536 维向量。

embeddingToBytes() 把它从 Java 的 float[] 变成原始字节数组——因为 Redis 不认识 Java 的 float，只认识原始字节。

用户问题: "布洛芬能和阿司匹林一起吃吗？" │ ▼ 千问 text-embedding-v3 API float[]{ 0.12, -0.34, 0.08, 0.41, 0.03, ... } // 1536 个数字 │ ▼ embeddingToBytes() — 每个 float 占 4 字节 byte[]{ 0x0A, 0xD7, 0x23, 0x3E, ... } // 6144 字节 │ ▼ 作为参数传入 Redis KNN 查询

4.3 构建 KNN 查询语句 L195-200

    // 构建 KNN 查询: "*=>[KNN K @embedding $BLOB AS score]"
    String query = "*=>[KNN " + maxResults + " @embedding $BLOB AS score]";
    FTSearchParams params = FTSearchParams.searchParams()
            .addParam("BLOB", queryVector)
            .dialect(2)
            .limit(0, maxResults);

这是 RediSearch 的 KNN 查询语法。我们逐块拆解：

片段含义
*=>[KNN 5 @embedding $BLOB AS score]完整查询语句
*"所有文档都参与"——不做前置过滤（如果只想搜 MEDICINE 分类，可以改成 @category:{MEDICINE}=>[KNN ...]）
KNN 5返回距离最近的 5 个结果（K=5）
@embedding在哪个字段上做向量搜索——对应存入时的 embedding 字段
$BLOB占位符，实际值通过 addParam("BLOB", queryVector) 传入——这是 RediSearch 的参数化查询，防止二进制数据破坏查询语法
AS score把距离值命名成 score，结果中可以通过 __score 获取
dialect(2)使用 RediSearch 查询方言版本 2——KNN 语法要求 dialect ≥ 2
limit(0, maxResults)偏移量 0，最多返回 5 条
为什么要把字节数组放在参数里而不是直接拼进查询字符串？
因为 embedding 是二进制数据（6144 字节），如果直接拼字符串，会出现不可打印字符，导致查询解析失败。$BLOB 占位符 + addParam 是 RediSearch 专门为二进制向量设计的传参方式。

4.5 ⭐ 解析结果（最核心部分）L204-228

        List<EmbeddingMatch<TextSegment>> matches = new ArrayList<>();

        for (Document doc : result.getDocuments()) {
            // ─── 第1步：分数转换 ───
            String scoreStr = getProperty(doc, "__score");
            double score = scoreStr != null
                    ? 1.0 - Double.parseDouble(scoreStr)
                    : 0.0;

                    ? blobToEmbedding(embeddingRaw)
                    : new float[0];

            TextSegment segment = TextSegment.from(
                    content != null ? content : "", metadata);

                    new Embedding(vector), segment));
        }

        return new EmbeddingSearchResult<>(matches);

步骤1：分数转换——从"距离"到"相似度"

这是最容易误解的地方！

KNN 搜索返回的是 距离（越小越相似），但我们的业务需要的是 相似度分数（越大越相似）。 Redis 使用 COSINE 距离，范围是 [0, 2]，0 = 完全同向，2 = 完全反向。

Redis 返回的 __score（距离） 我们想要的 score（相似度） COSINE 距离 = 0 ──→ 1.0 - 0 = 1.0 ✅ 最相似 COSINE 距离 = 0.3 ──→ 1.0 - 0.3 = 0.7 🟡 较相似 COSINE 距离 = 0.5 ──→ 1.0 - 0.5 = 0.5 🟠 不太像 COSINE 距离 = 0.8 ──→ 1.0 - 0.8 = 0.2 🔴 基本不相关（低于 0.65 被丢弃） COSINE 距离 = 1.0 ──→ 1.0 - 1.0 = 0.0 ❌ 完全无关（丢弃）
所以这一行代码的本质是： 
double score = 1.0 - Double.parseDouble(scoreStr);
把 Redis 的"距离越小越好"翻转成"分数越大越好"。

步骤2：提取原向量（可选）

float[] vector = blobToEmbedding(embeddingRaw);  // 字节 → float[]

从 Redis 返回的文档中提取 embedding 字段并反序列化。这个字段不一定需要（检索时主要用 content）， 但 LangChain4j 的 EmbeddingMatch 要求携带它，方便调用方做额外处理（比如 Reranker 二次排序）。

4.6 辅助方法

// 从 Jedis Document 中提取属性值
// Document.getProperties() 返回的是 Iterable<Map.Entry>，不是 Map！
// 所以必须遍历查找
private String getProperty(Document doc, String key) {
    for (Map.Entry<String, Object> entry : doc.getProperties()) {
        if (key.equals(entry.getKey())) {
            Object val = entry.getValue();
            return val != null ? val.toString() : null;
        }
    }
    return null;
}

// 把所有属性（除了 embedding 和 __score）转成 LangChain4j Metadata
private Metadata buildMetadata(Document doc) {
    Metadata metadata = new Metadata();
    for (Map.Entry<String, Object> entry : doc.getProperties()) {
        String key = entry.getKey();
        Object val = entry.getValue();
        if (val == null || "__score".equals(key) || "embedding".equals(key)) continue;
        metadata.put(key, val.toString());  // 全转字符串
    }
    return metadata;
}
踩坑提醒：Jedis 6.x 的 Document.getProperties() 返回 Iterable<Map.Entry>，不是 Map！所以不能用 .get("key") 直接取，必须遍历。这是 Jedis 6.0 的破坏性变更。

6. 索引管理（第 267-329 行）

Redis 的 RediSearch 模块就像给 Hash 数据建了一个"专业搜索引擎"。
没有索引 → 只能按 Key 精确查找 → 不能做 KNN 搜索
有索引 → 可以高效 KNN 搜索 → 毫秒级返回 Top-K 
// 索引 Schema 翻译成人话：
Schema schema = new Schema()
    .addTextField("content", 1.0)        // content → 可全文搜索，权重 1.0
    .addHNSWVectorField("embedding", ...) // embedding → HNSW 向量索引
    .addTagField("title")                // title → 精确匹配标签
    .addTagField("category")             // category → 可过滤: @category:{MEDICINE}
    .addNumericField("chunk_index")      // chunk_index → 数值范围查询
    .addNumericField("created_at")       // created_at → 可按时间过滤
    .addTagField("doc_id");              // doc_id → 删除时按文档批量定位

ensureIndexExists() 的策略：先尝试用 ftInfo() 查询索引是否存在，如果抛出异常就调用 createIndex() 创建。这避免了每次启动都重复创建。

7. 向量序列化 —— 编码/解码（第 331-368 行）

核心问题：Java 的 float[] 怎么存进 Redis，又怎么读回来？

写入：float[] → 字节 → RediSearch BLOB

// 步骤1: float[] → byte[]
private byte[] embeddingToBytes(Embedding embedding) {
    float[] vector = embedding.vector();               // 1536 个 float
    ByteBuffer buffer = ByteBuffer.allocate(vector.length * 4);  // 1536×4 = 6144 字节
    buffer.order(ByteOrder.LITTLE_ENDIAN);             // 小端字节序（RediSearch 要求）
    for (float v : vector) {
        buffer.putFloat(v);                            // 每个 float → 4 字节
    }
    return buffer.array();                             // 返回 6144 字节数组
}

// 步骤2: byte[] → String（ISO-8859-1 无损转换）
// 为什么不直接用 byte[]？因为 Jedis 的 hset 要求 Map<String, String>
private String bytesToBlob(byte[] bytes) {
    return new String(bytes, StandardCharsets.ISO_8859_1);
    // ISO-8859-1 的特点：每个字节和字符一一对应，无信息丢失
    // 如果用 UTF-8，多字节字符会被错误地合并/拆分
}

读取：RediSearch BLOB → 字节 → float[]

private float[] blobToEmbedding(String blob) {
    byte[] bytes = blob.getBytes(StandardCharsets.ISO_8859_1);  // 无损还原
    ByteBuffer buffer = ByteBuffer.wrap(bytes);
    buffer.order(ByteOrder.LITTLE_ENDIAN);
    float[] vector = new float[bytes.length / 4];    // 6144 / 4 = 1536
    for (int i = 0; i < vector.length; i++) {
        vector[i] = buffer.getFloat();               // 每 4 字节 → 1 个 float
    }
    return vector;
}

为什么必须用 ISO_8859_1（Latin-1）而不是 UTF-8？
UTF-8 是多字节编码：一个字符可能对应 1~4 个字节。当你把随机的 6144 字节用 UTF-8 解码再编码，可能丢失数据。
ISO-8859-1 是单字节编码：每个字节 0x00-0xFF 对应唯一字符，保证 100% 无损。
RediSearch 内部拿到 BLOB 后也不解析字符，直接用原始字节做 FLOAT32 向量运算。

9. 常见疑问 FAQ

Q1: 为什么 Redis 存了 1536 个数字，却占 6144 字节？

因为每个 float 在 IEEE 754 标准中占 4 个字节。1536 × 4 = 6144。如果后续切到 1024 维 embedding，占用就是 4096 字节。

Q2: KNN 和 SQL 的 WHERE 有什么区别？

SQL 的 WHERE 是精确匹配（category = 'MEDICINE'），KNN 是模糊匹配（"意思最接近的 5 个"）。两者可以组合使用。

Q3: minScore = 0.65 是怎么定的？

经验值。太低 → 无关内容混进来，污染 Prompt。太高 → 可能找不到任何结果。实际可以调参，不同场景可能不同。

Q4: 如果检索返回空怎么办？

RagService 有降级策略：检索失败或空结果时，直接用用户的个人数据（用药清单等）构造 Prompt，不注入知识库内容。

Q5: 为什么不直接用 MySQL 存向量？

MySQL 没有向量索引。用 MySQL 做 KNN 需要全表扫描 + 计算 1536 维距离，数据一多大爆炸。Redis Stack 有专门的 HNSW 索引，毫秒级。

Q6: HNSW 索引建好了，新增数据会自动加入吗？

是的。RediSearch 的索引是"后建式"——索引建好后，任何匹配 prefix 的新 Hash 写入都会自动被索引，无需手动操作。这是 Redis Stack 最大的优势之一。
