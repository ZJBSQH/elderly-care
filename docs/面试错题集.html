第1题：elderly-ai 模块架构定位

得分 2.5/5 第一轮 · 架构与设计 Java Spring

你的回答

elderly-ai模块是在微服务里面是对用药模块的扩充，用户通过ai模块对自己的用药情况更加了解并收到有更多的建议，它有一个redis充当下数据库，里面存储一些文档的字节数据，依赖remind、medicine、health模块的查询接口获取对应模块的数据，通过feign远程调用这些数据接口。

错误分析

  • 遗漏了 RAG 增强检索——这是 elderly-ai 最核心的差异化能力
  • 遗漏了知识库管理（KnowledgeBaseController：文档 CRUD + 资讯同步）
  • 依赖的服务说错了——你说了 health，但代码中 没有 HealthFeignClient，实际有的是 NewsFeignClient
  • Redis 定位不精准——不是"充当数据库"，而是 Redis Stack (RediSearch) 使用 HNSW 算法做向量存储与 KNN 语义检索

完整答案

elderly-ai 是老年护理云平台中的无状态聚合 AI 服务（端口 8087），没有独立数据库。

三大核心能力：
  1. 用药 AI 问答（AssistantController）：通过 Feign 聚合 medicine/remind 数据构建上下文，调用千问大模型流式生成回答
  2. RAG 增强检索问答（RagAssistantController）：用户问题 → Embedding 向量化 → Redis HNSW KNN 检索 → 增强 Prompt → 流式生成（含降级策略）
  3. 知识库管理 + 资讯同步（KnowledgeBaseController）：医学文档 CRUD，支持从 elderly-news 一键同步文章到向量库
依赖的下游服务：elderly-medicine、elderly-remind、elderly-news（通过 OpenFeign 调用，FeignConfig 自动转发 JWT Token）
存储：Redis Stack (RediSearch) 作为向量存储，使用 HNSW 索引 + COSINE 距离进行 KNN 语义检索。
AI 模型：阿里云 DashScope 千问 qwen3.6-flash（对话）+ text-embedding-v3（嵌入，1024 维）。
模块的完整定位要看三个 Controller 的职责分工，仅从 Feign 调用去理解会漏掉核心的 RAG 管道和知识库管理能力。

第2题：MedicalAssistant 无状态设计

得分 2.5/5 第一轮 · 架构与设计 LangChain4j @AiService

你的回答

因为这个项目比较小，直接通过context调用整个用户用药数据做上下文，已经全部覆盖了，不需要memory功能，这样可以保证ai接收到所有数据，缺点是当上下文过大时整个上下文会多许多无用数据，导致token的转化率不高，这个功能只是用来做保底措施的，当rag功能无法使用时，使用调取上下文的加medicalAssistant的定义，临时代替使用rag功能。

错误分析

  • "项目比较小"不是真正的设计理由——核心原因是 数据的实时性：用药记录和提醒任务在持续变化，缓存的 Conversation Memory 会使用过时数据
  • 遗漏了"简单性"这个关键优点：不需要配置 ChatMemoryProvider、不需要管理会话生命周期、每次调用完全独立、天然支持水平扩展
  • 定位描述不准确：MedicalAssistant 不是 "RAG 的临时替代品"——它在 RagService 中也是核心执行引擎（正常流程和降级流程最终都调用 assistant.chat()）

完整答案

不使用 @MemoryId 的根本原因：每次请求的用药数据、提醒等上下文 必须实时重新构建——药品清单可能被修改、服药记录随时更新、提醒任务状态在变化。如果缓存了历史对话 Memory，AI 会基于过时数据回答，这在医疗场景下是危险的。

无状态设计的优点：
  1. 每次调用完全独立，上下文由调用方显式控制
  2. 不需要配置 ChatMemoryProvider 和 ChatMemoryStore
  3. 不需要管理会话的生命周期、过期、清理
  4. 天然支持水平扩展——任意请求可以落到任意实例
MedicalAssistant 的定位：统一的 AI 出口，而非 RAG 的"备用方案"： 
// RagService.java — 正常流程（走 RAG）：
String augmentedPrompt = buildAugmentedPrompt(relevantDocs, personalContext, question);
return assistant.chat(augmentedPrompt);  // ← 同样调用 assistant

// RagService.java — 降级流程（RAG 失败时）：
return assistant.chat(fallbackContext + "\n\n用户问题: " + question);  // ← 也是 assistant

第4题：安全配置矛盾

得分 2.5/5 第一轮 · 架构与设计 Security JWT

你的回答

未登录的用户会直接在elderId为1的账号的背景下进行对应的ai使用，会导致该用户的信息泄露，这是一个测试编写，在一个独立的theme文件中测试rag功能的，后续开发结束会修改。

错误分析

  • 没有识别出架构级矛盾：permitAll 全部放行 → JWT 过滤器不解析 Token → SecurityUtil.getCurrentUserId() 永远返回 null → 永远走 elderId=1 兜底
  • 遗漏了知识库管理接口的安全漏洞：KnowledgeBaseController 路由 /ai/kb/** 也在白名单内，任何人都可以上传/删除/修改知识库
  • 没有提到缺少技术债务标记：代码中没有任何 TODO/FIXME 标记

完整答案

三个层次的问题：
  1. 逻辑自相矛盾：permitAll("/ai/**") 告诉 Spring Security "所有 AI 路径都不需要 JWT 认证"。即使前端带了 Authorization 头，JwtAuthenticationFilter 也会直接放行不解析，SecurityContext 中永远不会有 Authentication 对象，SecurityUtil.getCurrentUserId() 永远返回 null，elderId=1 的兜底分支成为 唯一可行的路径。
  2. 知识库管理接口完全暴露：KnowledgeBaseController 的路由是 /ai/kb/**，也在白名单内。意味着任何人都可以： 
    # 不需要任何认证即可操作知识库
POST   /ai/kb/document          # 上传文档
DELETE /ai/kb/document/{id}     # 删除文档
PUT    /ai/kb/document/{id}     # 修改文档
POST   /ai/kb/sync-from-news    # 触发资讯同步
  3. 缺少技术债务追踪：代码里没有 TODO/FIXME/@Deprecated 标记。
正确的修复方向：

        // 其余接口全部需要认证
    };
}

第5题：RedisEmbeddingStore KNN 检索

得分 1.5/8 第二轮 · 核心实现 Redis Stack HNSW

你的回答

1. embeddingModel提供，参数通过@embedding声明传递。
2. 为了获得向量的相似度，原始值代表当前向量的值。
3. 不清楚。

错误分析

  • 第1问完全错误：KNN 语法是 Redis Stack (RediSearch) 的查询语法，不是 embeddingModel 提供的
  • $BLOB 参数传递：通过 FTSearchParams.addParam("BLOB", queryVector) 绑定，不是"声明传递"
  • Score 转换理解不够：原始值是 COSINE 距离（范围 [0,2]），需要 1.0 - 距离 转为相似度
  • 第3问未回答

完整答案

1. KNN 查询语法与参数传递： 
// RediSearch 的 KNN 查询语法（非 embeddingModel 提供）
String query = "*=>[KNN " + maxResults + " @embedding $BLOB AS score]";

// $BLOB 是参数占位符，通过 FTSearchParams 绑定实际值
FTSearchParams params = FTSearchParams.searchParams()
    .addParam("BLOB", queryVector)   // ← 在这里绑定：占位符名 → 字节数组
    .dialect(2)                      // ← RediSearch 方言版本
    .limit(0, maxResults);

// @embedding 是索引中向量字段的名称（创建索引时定义的 HNSW 向量字段）
2. Score 转换： 
// 索引创建时指定：DISTANCE_METRIC COSINE
// COSINE 距离范围 [0, 2]：0 = 完全相同，1 = 正交，2 = 完全相反
// 转换公式：相似度 = 1.0 - COSINE距离
//   距离 0.0 → 相似度 1.0（最相关）
//   距离 0.35 → 相似度 0.65（刚好达到阈值）
//   距离 2.0 → 相似度 -1.0（被过滤）

double score = scoreStr != null ? 1.0 - Double.parseDouble(scoreStr) : 0.0;
if (score < minScore) continue;  // minScore 默认 0.65
3. 检索异常的处理策略与潜在问题：
  1. 静默失败——catch 后只打 log 返回空列表，调用方（RagService）无法区分"没有匹配结果"和"Redis 挂了"
  2. Score 转换有精度风险——Double.parseDouble(scoreStr) 可能抛 NumberFormatException，没有被单独 try-catch
  3. 没有区分异常类型——索引不存在需要重建 vs 连接失败需要重试 vs 语法错误需要告警，三种场景处理策略完全不同
  4. 降级路径不一致——search() 失败返回空列表，embeddingModel.embed() 失败抛异常，两种失败的降级行为不统一

改进建议（代码示例）

@Override
public EmbeddingSearchResult<TextSegment> search(EmbeddingSearchRequest request) {
    try {
        // ... 执行检索 ...
        return new EmbeddingSearchResult<>(matches);
    } catch (redis.clients.jedis.exceptions.JedisConnectionException e) {
        // 连接失败：应重试或明确向上抛异常让调用方走降级
        log.error("Redis 连接失败，无法执行向量检索", e);
        throw new RuntimeException("向量检索服务不可用", e);
    } catch (redis.clients.jedis.exceptions.JedisDataException e) {
        // 索引不存在：尝试重建
        log.warn("索引 {} 可能不存在，尝试重建", indexName);
        ensureIndexExists();
        return new EmbeddingSearchResult<>(List.of());
    } catch (NumberFormatException e) {
        log.error("Score 解析失败: {}", e.getMessage());
        return new EmbeddingSearchResult<>(List.of());
    }
}

第6题：IngestionPipeline 文档摄取管道

得分 3.5/8 第二轮 · 核心实现 Embedding Chunking

部分错误

使用递归字符切分，先按段落切分，段落长再按句子切分，这样医学文档中，一些总结的段落是能够保存的，段落长的话一般是功能详细描述，通过重叠字数衔接也能保证一下句子不会断关联，分批交，当多线程调用时，片段之间会发生错乱。

错误分析

  • 第1问漏了具体参数（maxChunkSize=500, maxOverlap=50）和医学文档的结构化短句弱点
  • 第2问错误严重——embedAll() 不是分批，是一次性提交；且"多线程片段错乱"是错的（不存在并发）

完整答案

1. 递归切分策略与医学文档适配： 
// 配置参数
elderly.rag.chunk.max-size: 500     // 最大 500 字符
elderly.rag.chunk.max-overlap: 50   // 相邻重叠 50 字符

// 代码实现
var splitter = DocumentSplitters.recursive(maxChunkSize, maxOverlap);
List<TextSegment> segments = splitter.split(Document.from(text));

// 切分策略：段落 → 句子 → 字符（递归降级）
// 优点：总结性段落完整保留
// 弱点：药品说明书的结构化短句（如"【用法用量】一次2粒，一日3次"）
//       一个 500 字符的 chunk 可能混杂多个不相关的药理学主题
2. embedAll() 的真实行为： 
// embedAll() 是一次性同步提交，不是分批！
List<Embedding> embeddings = embeddingModel.embedAll(segments).content();
// 100 个片段 = 1 个 HTTP 请求发给 DashScope API

// 风险：
// ① 超时风险：DashScope embedding API 超时 30s，100 个片段可能超时
// ② 部分失败无法恢复：没有断点续传，不知道哪几个片段成功了
// ③ 无重试机制：整个 ingest() 没有 @Retryable，一次失败全丢

// 改进方案：分批发送
private List<Embedding> embedBatch(List<TextSegment> segments, int batchSize) {
    List<Embedding> all = new ArrayList<>();
    for (int i = 0; i < segments.size(); i += batchSize) {
        int end = Math.min(i + batchSize, segments.size());
        List<TextSegment> batch = segments.subList(i, end);
        all.addAll(embeddingModel.embedAll(batch).content());
        log.info("嵌入进度: {}/{}", end, segments.size());
    }
    return all;
}

第8题：RagService 双层异常处理

得分 1.5/7 第二轮 · 核心实现 Exception RAG

你的回答

都是在rag失效的情况下，捕获到retrieveRelevant的异常处理，因为askWithRag里面调用了retrieveRelevant，被程序异常接受，不合理，因为这时候应该使用降级处理。

错误分析

  • 没有区分两层 catch 各自覆盖的不同异常来源
  • 没有发现内层 catch 吞异常导致外层 catch 永远捕获不到 search 的异常
  • 降级逻辑的"行为不一致"没有被识别——embed 失败走降级，search 失败不走降级

完整答案

两层 catch 的异常范围分析：
askWithRag() 外层 try-catch（可捕获 embed 异常 ✅） └─ retrieveRelevant() ├─ embeddingModel.embed(question) ← 抛异常 → 不被内层捕获 → 穿透到外层 ✅ 走降级 └─ embeddingStore.search(request) └─ 内层 try-catch ← 抛异常 → 被内层吞掉 → 返回空列表 ❌ 不走降级

关键发现：embeddingStore.search() 内部已经 catch 了所有异常并返回空列表，从不向外层抛异常。因此 askWithRag() 的外层 catch 只能捕获到 embeddingModel.embed() 的异常。

而 search() 失败时，外层 catch 完全感知不到——代码拿着"空的参考知识"继续组装 Prompt，用户收到的回答没有知识库增强，但完全不知道检索失败了。

同一个方法内行为不一致：embed 失败 → 有降级兜底（走 fallbackContext），search 失败 → 静默返回空列表（无降级）。这是设计缺陷。

修复方案

// 方案：让 retrieveRelevant 统一抛异常，由外层统一降级
private List<EmbeddingMatch<TextSegment>> retrieveRelevant(String question) {
    // embed 异常直接向上抛
    Embedding queryEmbedding = embeddingModel.embed(question).content();
    EmbeddingSearchRequest request = EmbeddingSearchRequest.builder()
            .queryEmbedding(queryEmbedding)
            .maxResults(maxResults)
            .minScore(minScore)
            .build();
    // search 异常也向上抛（不再内部吞掉）
    EmbeddingSearchResult<TextSegment> result = embeddingStore.search(request);
    return result.matches();
}
// 外层 askWithRag() 的 try-catch 统一处理所有降级

第9题：LangChain4j Flux 与 Reactor 桥接

得分 1/5 第三轮 · 技术栈与集成 LangChain4j Reactor

你的回答

第1问不清楚；第2问是实现帮忙实现流式输出。

完整答案

1. 返回 Flux<String> 的底层机制：
LangChain4j 的 AIServiceFactory 代理在运行时检测 @AiService 接口方法的返回类型：
  • 返回 String → 自动注入 chat-model bean（普通同步调用）
  • 返回 Flux<String> / Flowable<String> 等 → 自动注入 streaming-chat-model bean（流式调用）
Flux 返回类型本身就是"路由开关"——不需要任何显式配置或注解来区分。框架根据方法签名自动选择使用哪个模型 bean。

2. langchain4j-reactor 的真正作用——桥接：
LangChain4j 原生的流式 API 是基于回调的 StreamingResponseHandler： 
// LangChain4j 原生回调风格
streamingChatModel.generate(userMessage, new StreamingResponseHandler<AiMessage>() {
    @Override public void onNext(String token) { /* 处理每个 token */ }
    @Override public void onComplete(Response<AiMessage> response) { /* 完成 */ }
    @Override public void onError(Throwable error) { /* 错误 */ }
});
而 Spring WebFlux 需要 Flux<String>。

langchain4j-reactor 提供了适配层（ReactorStreamingResponseHandler），将回调风格转为 Reactive Streams 的 Flux，使得 @AiService 接口可以直接返回 Flux<String>。

没有这个依赖，@AiService 接口无法返回 Flux<String>。

第10题：Flux.defer() 的真实作用

得分 1/5 第三轮 · 技术栈与集成 WebFlux Reactor

你的回答

失去流式输出的效果，如果context出现问题，那么这个message就会出现上下文对应不上问题，导致回答准确性下降。

错误分析

  • 方向完全错误——不是"失去流式输出"
  • 真正的问题是 阻塞 Netty 事件循环线程（Event Loop Blocking）

完整答案

真正的问题：Event Loop Blocking 
// ❌ 没有 Flux.defer()：阻塞了 Netty 事件循环线程
@PostMapping(value = "/medicine/question", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> processMedicineQuestion(@RequestBody String question) {
    Integer elderId = securityUtil.getCurrentUserId();
    // ⚠️ 这行在请求处理线程（Netty Event Loop）上立即执行
    // buildAndFormat() 内部是同步 Feign 调用 —— 阻塞 IO！
    String context = contextBuilder.buildAndFormat(elderId);  // 可能阻塞 2-3 秒
    return assistant.chat(context + "\n\n用户问题: " + question)...;
}

// ✅ 使用 Flux.defer()：延迟到订阅时执行
return Flux.defer(() -> {
    // defer 确保阻塞操作在订阅线程（Scheduler）上执行
    // 不影响 Netty Event Loop 处理其他请求
    String context = contextBuilder.buildAndFormat(elderId);
    return assistant.chat(context + "\n\n用户问题: " + question);
})
核心原理：
WebFlux 的请求处理运行在 Netty Event Loop 线程上，线程数很少（通常 = CPU 核数）。
buildAndFormat() 内部是同步阻塞的 Feign HTTP 调用（阻塞 IO），如果在 Event Loop 线程上执行，会阻塞整个线程几秒钟，导致其他请求排队等待 → 服务吞吐量暴跌。
Flux.defer() 将阻塞操作延迟到有人订阅 Flux 时，WebFlux 会在 Scheduler 线程上执行订阅代码，Event Loop 线程立即释放去处理下一个请求。

第11题：SSE 格式一致性

得分 0/5 第三轮 · 技术栈与集成 SSE

你的回答

完整答案

两者的关键区别——[DONE] 结束事件：
端点返回格式是否有 [DONE] 事件
AssistantController
/ai/medicine/question		 token.map(data -> SSE) ❌ 无结束标记
AssistantController
/ai/medicine/today-advice		 token.map(data -> SSE) ❌ 无结束标记
RagAssistantController
/ai/rag/ask		 token.map(data -> SSE) + [DONE] ✅ 有结束标记
RagAssistantController
/ai/rag/health-knowledge		 token.map(data -> SSE) + [DONE] ✅ 有结束标记

这是一个一致性问题——属于遗漏： 
// AssistantController — 没有 [DONE] 事件
return Flux.defer(() -> { ... return assistant.chat(fullMessage); })
    .map(token -> ServerSentEvent.<String>builder()
            .data(token)
            .build());
// ↑ 流结束后没有通知前端

// RagAssistantController — 有 [DONE] 事件
return ragService.askWithRag(elderId, dto.getQuestion())
    .map(token -> ServerSentEvent.<String>builder()
            .data(token)
            .build())
    .concatWith(Mono.just(  // ← 显式拼接结束事件
            ServerSentEvent.<String>builder()
                    .data("[DONE]")
                    .build()));
如果前端（elder_care/pages/ai/chat.vue）依赖 [DONE] 来判断流结束（比如：收到 [DONE] → 隐藏加载动画 → 标记消息完成），那么 AssistantController 的流永远不会触发这个逻辑，前端可能一直显示"加载中"。

修复：统一在 AssistantController 中也拼接 [DONE] 事件，或者统一都不加，由前端监听流的 complete 事件（而非 [DONE] 标记）。无论哪种方案，关键是保持一致性。

public String uploadDocument(KnowledgeDocumentDTO dto) {

public List<KnowledgeDocVO> listDocuments(String keyword) {
    Set<String> docIds = jedis.smembers("rag:doc:ids");
    List<KnowledgeDocVO> vos = new ArrayList<>();
    for (String docId : docIds) {
        Map<String, String> meta = jedis.hgetAll("rag:doc:" + docId + ":meta");
        if (meta.isEmpty()) continue;

        if (keyword != null && !meta.get("title").contains(keyword)) continue;

            .id(docId)
            .title(meta.get("title"))
            .category(meta.get("category"))
            .chunkCount(Integer.valueOf(meta.get("chunk_count")))
            .build());
    }
    return vos;
}

temperature = 0.4 在医疗场景下是否合适？

✅ 合适，甚至可以考虑更低（0.1-0.3）
Temperature特性适用场景
0.0 - 0.3确定性输出，每次回答几乎一致医疗建议、法律咨询、代码生成
0.4 - 0.7有一定灵活性，输出有变化对话、客服、内容总结
0.8 - 1.2高创造性，输出不可预测创意写作、头脑风暴

为什么医疗场景需要低 temperature：
  1. 安全性优先——医疗信息需要准确、一致。高 temperature 意味着相同问题可能得到不同的答案，在用药建议场景下是危险的
  2. 事实性 > 创造性——用户需要的是可靠的医学知识复述，不是"创新"的回答
  3. 与 RAG 搭配有意义——RAG 已经给定了参考知识，低 temperature 让模型更贴近检索结果而非自由发挥，有效抑制幻觉
